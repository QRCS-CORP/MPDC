37/*
38* Client to Server:
39* On internal network, client could have the RDS public verification key.
40* On external network, how would the client get that key?
41* How would browsers authenticate websites for example?
42* This is done using IDGs, which are part of the ISPs infrastructure.
43* An IDG caches RDS certificates of other ISPs, which in turn sign domain certificates
44* of domain operators on their network.
45* IDGs keep caches of ISP level certificates, and are able authenticate or forward an IDGs public signature key.
46* The ISPs charge the domain operators for signing their certificates, which can include reliability metrics
47* to create a dynamic pricing model e.g. phishing sites can be downgraded in the reliability model,
48* which translates to how browsers and other softwares apply permissions to a website.
49* Upstream ISPs can negotiate IDG database contracts in a similar way that BGP is done,
50* Conditional peering arrangements, in which a sliding scal cost structure is applied depending on
51* request volume and the relaibility of the traffic.
52* Downstream ISPs peer to upstream ISPs, from which they can do recursive authentication queries,
53* and can also be 'taxed' based on their reliability metrics.
54* ISPs charge client domains for the security service as part of their pricing model just like DNS,
55* They sign certificates, guaranteeing the ownership of domains they host, and are responsible for
56* the reliability metric applied to that certificate which is generated in a dynamic model.
57* The ISP can downgrade reliability, which in turn can interact with software at the user level,
58* applying or removing permissions based on the trust metric of a domain.
59* Likewise, traffic that is rejected by a local IDG, can reduce the trust metric of the remote domain via
60* the IDG, which caches the certificate. This distributes control over how clients on the network perceive the trust,
61* while applying the same controls to the domain host.
62* An economic cost can be applied, via the trust relationships between ISPs, and ISP and domain,
63* using 'living contracts' in which the cost of maintaining the peer relationship goes up for bad actors,
64* or for the ISPs that host them.
65* This shifts authentication reponsibility onto the upstream ISPs, which contract IP space and bandwidth
66* to smaller ISPs, and in turn, downstream ISPs and the domains they host.
67* Downstream ISPs that host bad actors would see the cost of their access increase, and can apply that cost
68* to the bad domains, or revoke their access.
69* A host connects to a website and receives a certificate from a domain it does not know.
70* It queries the domain level IDG, to see if the gateway has this certificate cached.
71* If the domain level IDG does not know this domain, it forwards the query to the ISP level IDG.
72* The ISP level IDG may be a downstream ISP, if it does not know the certificate, it forwards the query to the upstream ISP.
73* If the upstream ISP does not know the certificate, or the ISP that issued it, it queries other ISPs that it has a relationship with.
74* If the upstream ISP knows the issuer, or knows of the issuers upstream ISP, then it verifies the certificate with them.
75* This can be done using hints; the serial number of the certificate contains 4 128-bit identifiers:
76* The certificate serial number, a unique number associated with that certificate.
77* The domain level serial number, which is the serial number of the domains certificate.
78* The hosting ISP serial number, which is the ISPs certificate serial number.
79* The upstream ISPs serial number, this is the guarantor of the ISP hosting the domain.
80* ISPs can have multiple certificates, as can domains, and the serial number is a composite of the unique identifier of the certificate,
81* and an address identifier, indicating which entity owns the certificate.
82* domain level id: 24-bits identify the upstream ISP, 24-bits identify the hosting ISP, 24-bits identify the domain,
83* 16 bits identify the server, and the remaining 16-bits identify the certificate itself.
84*
